Concept · Security Architecture
Tokenless Gateway
Agents never touch API keys. All LLM provider communication flows through a centralized trust boundary. Identity-based authentication, not token-based.
The Problem

API keys in agent code means leaked secrets, per-agent billing, no audit trail. Every agent embedding its own key is a security nightmare. One compromised agent exposes every provider credential. There is no visibility into who called what, when, or why.

The Solution

The gateway owns all secrets. Agents authenticate by identity, not by token.

How it works
Agent Request
Agent sends task via br invoke — no API key needed. The agent identifies itself, describes the task, and the gateway handles the rest.
Gateway Routing
Gateway selects optimal provider based on task type and agent permissions. Permission matrix defines which agents can access which providers.
Provider Call
Gateway adds credentials, calls provider, returns result to agent. Full audit trail. Every call logged. Zero secrets in agent code.
Verification

verify-tokenless-agents.sh runs on every commit. Scans all agent code for leaked tokens. Zero tolerance. If a key appears in agent code, the build fails. No exceptions.
