Role modes
==========

This file explains how different roles should use the Canonical hierarchy.
The structure stays the same, but the emphasis changes by role.

Role 1: Human creator
---------------------
Primary concern:
Creating, shaping, and advancing real items.

Recommended behavior:
- Start with `01-QUICKSTART.txt`
- Use `TEMPLATE-ITEM.txt`
- Keep metadata truthful
- Move items steadily through workflow states
- Use `07-RECIPES.txt` when unsure

Typical focus:
- clarity of intent
- accurate item content
- meaningful handoff

Role 2: Operator
----------------
Primary concern:
Maintaining flow and preventing structural drift.

Recommended behavior:
- Use `11-CHECKLISTS.txt` before moves and handoffs
- Prefer `move-item.sh` over manual movement
- Keep `24-Conveyor-Belt` and `25-Transfer` visible
- Preserve traceability when it matters
- Watch for anti-patterns

Typical focus:
- state accuracy
- safe movement
- consistency across layers

Role 3: Agent
-------------
Primary concern:
Automating movement, updates, and documentation safely.

Recommended behavior:
- Treat the HOW-TO hub as the canonical manual
- Follow shell-safe naming rules
- Reuse existing workflow states instead of inventing new ones
- Update metadata whenever moving files
- Extend docs when recurring patterns appear

Typical focus:
- repeatability
- automation safety
- documentation discipline

Role 4: Auditor
---------------
Primary concern:
Understanding lineage, correctness, and trustworthiness.

Recommended behavior:
- Inspect `Current Step`, `Current Path`, and `Source`
- Compare item contents to their folder locations
- Read `EXAMPLE-TRACE.txt` and trace files when available
- Check whether handoffs are visible and documented
- Watch for duplicate clutter and archive misuse

Typical focus:
- audit trail quality
- structural integrity
- reference quality

Role 5: Architect
-----------------
Primary concern:
Changing the system without breaking its logic.

Recommended behavior:
- Read `09-DESIGN-PRINCIPLES.txt` first
- Avoid changing taxonomy for process reasons
- Preserve the shared 27-step motion model
- Improve by composition, not special cases
- Update `08-GLOSSARY.txt` and the HOW-TO index when meaning changes

Typical focus:
- model coherence
- long-term scale
- teachability

Role 6: Teacher
---------------
Primary concern:
Helping new users learn the system quickly.

Recommended behavior:
- Start with examples before theory
- Use the demo item and `EXAMPLE-TRACE.txt`
- Point learners to `01-QUICKSTART.txt`, `07-RECIPES.txt`, and `11-CHECKLISTS.txt`
- Explain taxonomy and workflow as separate ideas
- Reinforce the glossary language

Typical focus:
- onboarding clarity
- shared vocabulary
- practical understanding

Shared rule across all roles
----------------------------
The role changes the emphasis, not the structure.
Everyone uses the same layers, the same 27 steps, and the same canonical vocabulary.
