Every agent in the BlackRoad OS mesh needs to remember what it was doing. Not just at runtime — across reboots, across cluster migrations, across power failures on the Pi 5 nodes. Traditional approaches reach for Redis or SQLite. We tried both. Neither was the right abstraction.

The Problem With Mutable State

Key-value stores like Redis are fast and ergonomic, but they're inherently mutable. An agent writes its state, the node crashes, and you're left asking: was that write committed? You add persistence flags. Then you add AOF logging. Then you're basically reimplementing a journal on top of a store that wasn't designed to be one.

SQLite is better — it's ACID-compliant and file-backed. But it's still row-oriented, which means deletes and updates destroy history. For agents, history is the state. You can't diff two snapshots if you only kept the latest.

— constraint
Agent memory must survive: power loss, node migration, concurrent writes from sub-agents, and rollback requests from the orchestration layer. Mutable stores fail at least two of these.

PS-SHA∞: Append-Only by Design

PS-SHA∞ is a cryptographically-chained append-only journal. Every memory write is a new entry that includes the SHA hash of the previous entry. You get an immutable, verifiable chain where any tampering is detectable and the full history is always recoverable.

journal entry schema
# Each entry in the PS-SHA∞ journal
{
  "seq":       1024,
  "ts":        "2025-11-14T03:22:11Z",
  "agent_id":  "cecilia-01",
  "payload":   { … },
  "prev_hash": "sha256:9f3a…",
  "hash":      "sha256:4b7c…"
}

The prev_hash field is the key. It creates a hard dependency between entries — you cannot insert, delete, or reorder without breaking the chain. This makes the journal self-auditing.

Write path

Writes are sequential and synchronous at the journal level, but agents interact through a NATS event bus message queue, so concurrency is handled upstream. The journal itself never sees a race condition.

write path (python)
def commit(agent_id: str, payload: dict) -> JournalEntry:
    prev = journal.tail(agent_id)
    entry = JournalEntry(
        seq       = prev.seq + 1,
        ts        = utcnow(),
        agent_id  = agent_id,
        payload   = payload,
        prev_hash = prev.hash,
    )
    entry.hash = sha256(entry.canonical_bytes())
    journal.append(entry)
    return entry

Recovery Across Power Cycles

On boot, each agent reads its journal tail and resumes from the last committed entry. Because partial writes leave a broken hash chain, they're automatically detected and truncated. No recovery scripts. No integrity checks that themselves need to be trusted.

The journal doesn't need a separate integrity layer because integrity is structural. A broken chain is immediately visible. A valid chain is immediately trustworthy.

Tradeoffs

  • Journals grow forever — we compact on a 30-day rolling window, archiving cold entries to R2
  • Read performance requires indexing — we maintain a separate seq→hash index in SQLite
  • Cross-agent reads require a merge strategy — currently earliest-wins for conflicts
