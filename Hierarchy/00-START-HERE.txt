Hierarchy workspace
===================

Purpose
-------
This folder now contains two parallel views:

1. Legacy/
   A preserved copy of the original concept chain.

2. Canonical/
   A normalized operating model with ordered layers and a 27-step atomic workflow inside each layer.

Recommended path
----------------
Use the Canonical tree for new work.

How it works
------------
The system combines two ideas:
- taxonomy: the layers of meaning
- workflow: the local state machine inside each layer

Each layer now owns the same 27-step workflow.
That means every layer can handle intake, refinement, movement, and reference with the same smart and intricate conveyor logic.

Meaning
-------
An item can now move through a much finer chain before it leaves a layer.
The conveyor belt is step 24, not the whole system.
The system is designed for atomic progress and clean handoff.

Automation
----------
Use `Canonical/move-item.sh` to move or copy an item between workflow steps.
Use `--copy` when you want a downstream handoff but also want to preserve the upstream file as an audit trail.
Use `./start.sh` at the root of `/Applications/Hierarchy` as the front door for list, map, browse, where, what, why, search/find, run, show, go, status, trace, report, fix, move, review, daily, weekly, monthly, docs/guide/open, create/new, help, and doctor flows. Use `./start.sh browse layers` when you want the numbered taxonomy chain itself, `./start.sh browse workflow` when you want the shared 27-step state model itself, `./start.sh browse lifecycle` when you want whole journey maps, `./start.sh browse bootstrap` when you want the learning sequence, `./start.sh browse glossary` when you want canonical vocabulary, `./start.sh browse decisions` when you want judgment help, `./start.sh browse principles` when you want design values, `./start.sh browse anti-patterns` when you want common mistakes, `./start.sh browse checklists` when you want operator safeguards, `./start.sh browse troubleshooting` when you want fast fixes, `./start.sh browse roles` when you want role-specific emphasis, `./start.sh browse rhythms` when you want recurring operating timing, `./start.sh browse examples` when you want reusable patterns, and `./start.sh browse recipes` when you want practical playbooks. Use `./start.sh show search <query>`, `./start.sh show trace <filename-or-item-id>`, `./start.sh show what <topic>`, or `./start.sh show why <topic>` when you want the read-only umbrella form of the inspection and explanation layer. Use `./start.sh open browse`, `./start.sh open reports`, or `./start.sh open what` when you want `open` to act like the front door for root-native surfaces instead of only guide files. Use `./start.sh doctor`, `./start.sh move archive <item-file>`, `./start.sh move handoff <item-file> <target-workflow-dir>`, `./start.sh fix integrity`, or `./start.sh fix repair <item-file>` when you already know the exact operational action you want.
